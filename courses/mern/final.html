Sujet d'Examen : Application CRUD pour Librairie

Contexte et Objectifs

Vous devez implémenter une application CRUD destinée à la gestion d'une librairie. L'application permettra de créer, lire, mettre à jour et supprimer des livres via une API. Le serveur doit TOUJOURS répondre en JSON.

Stack Technologique

  • Backend :
    • Node.js, Express, MongoDB
    • Modules : express, cors
  • Frontend :
    • React, react-router-dom (création du projet avec Vite)
    • Bootstrap pour le design (l'apparence du front est laissée à l'appréciation des étudiants, tant que l'interface reste agréable)
  • Conteneurisation :
    • Utilisation de Docker avec un Dockerfile dédié pour le frontend et un pour le backend
    • Pas de docker-compose attendu

Endpoints de l'API

  • /api/books
    • GET : Récupérer la liste des livres.
    • POST : Ajouter un nouveau livre.
  • /api/books/:id
    • GET : Récupérer les détails d'un livre spécifique.
    • PUT : Mettre à jour un livre existant.
    • DELETE : Supprimer un livre.
  • /status
    • GET : Retourner le statut du serveur (par exemple, vérifier la connexion à la base de données).

Spécifications Techniques et Exigences

Modèle de Données

Chaque livre doit inclure les propriétés suivantes :

  • titre
  • auteur
  • ISBN

Gestion des Erreurs et des Réponses

  • Utiliser response.status(code_de_retour) pour renvoyer les codes HTTP appropriés (par exemple, 200 OK, 201 Created, 400 Bad Request, 404 Not Found, etc.).
  • Les codes de retour doivent être cohérents avec les standards HTTP.

Validation

  • Validation de base : vérifier la présence des champs requis (titre, auteur, ISBN) lors des opérations POST et PUT.
  • Validation de l'ID : s'assurer que l'ID fourni dans l'URL pour les requêtes paramétrées est valide.

Configuration et Déploiement

  • Backend :
    • Le serveur doit écouter par défaut sur le port 3000.
    • Une URL de base pour la connexion à la base de données doit être définie.
    • La gestion minimale des CORS doit être assurée pour permettre les requêtes depuis le frontend.
  • Documentation :
    • Le fichier README doit inclure :
      • Une description détaillée de l'API (endpoints, méthodes HTTP, paramètres, codes de réponse, etc.).
      • Les instructions pour le déploiement en local (sans nécessiter de script run.sh).

Organisation du Projet

  • Deux dossiers principaux :
    • frontend : contenant le code du front-end (sans le dossier node_modules)
    • backend : contenant le code du back-end (sans le dossier node_modules)

Autres Précisions

  • Docker : Chaque partie (frontend et backend) doit posséder son propre Dockerfile.
  • Pas de configuration spécifique pour différents environnements (développement, production, etc.).
  • Bootstrap doit être utilisé pour l'ensemble de l'application.
