Docker-K8s

Final Project: Multi-Container Web Application Deployment

1. Objective

  • Develop and deploy a multi-container web application with Docker Compose for local development and Kubernetes for production-like orchestration.
  • Configure both environments to make the application accessible at http://localhost:8777.

2. Application Details

  • Frontend Service:

    • Serves as the user-facing interface for interacting with the backend service.
    • Proxies requests to the backend for:
      • /write/{data}: Writes data to the backend.
      • /read: Reads all data written to the backend log file.
      • /compute: Simulates a heavy computation workload on the backend.
      • /status: Returns a JSON with backend health and status.
    • Exposes the application on port 3403.
    • Accessible via http://localhost:8777.

  • Backend Service:

    • Handles application logic, including:
      • Writing data to a log file (/write/{data}).
      • Reading data from the log file (/read).
      • Simulating heavy computation (/compute).
      • Providing a status endpoint (/status).
    • Stores data persistently in a volume (app-data) bound to ${PWD}/app-data on the host machine.
    • Exposes the application on port 3402.

Example /compute Route Code:

import time
import math

@app.route("/compute")
def compute():
    start_time = time.time()
    result = 0
    for i in range(1, 10000000):
        result += math.sin(i) * math.cos(i)
    end_time = time.time()
    execution_time = end_time - start_time
    return f"Computation result: {result}, executed in {execution_time:.2f} seconds\n"

3. Tasks

Docker Compose

  1. Create Dockerfiles:

    • Dockerfile for frontend (front-image).
    • Dockerfile for backend (back-image).

  2. Create a compose.yaml File:

    • Deploy front-container and back-container.
    • Map the frontend to http://localhost:8777.
    • Persist backend data using ${PWD}/app-data.

Kubernetes

  1. Create Kubernetes Manifests:

    • Deployment:
      • Deploy frontend and backend as separate deployments.
      • Pull container images from the local Docker registry (localhost:5000).
    • Services:
      • Use ClusterIP for internal communication between frontend and backend.
      • Expose the frontend service at http://localhost:8777 using a NodePort or LoadBalancer.
    • Volume:
      • Use a PersistentVolume (PV) and PersistentVolumeClaim (PVC) for backend storage (app-data).

  2. Configure Horizontal Pod Autoscaler (HPA):

    • Configure HPA to trigger scaling based on CPU usage (e.g., traffic to /compute).
    • Set minimum replicas to 1 and maximum replicas to 5.

  3. Perform a Load Test:

    • Simulate high traffic to both /write/{data} and /compute.
    • Observe scaling behavior using the Kubernetes Horizontal Pod Autoscaler (HPA).

4. Deliverables

  1. Docker Artifacts:

    • Dockerfiles for both frontend and backend.
    • A working compose.yaml file.

  2. Kubernetes Artifacts:

    • Manifests for:
      • Deployment for frontend and backend, pulling images from localhost:5000.
      • Services to expose the application and allow internal communication.
      • PersistentVolume and PersistentVolumeClaim for backend data storage.
      • Horizontal Pod Autoscaler configuration.

Here is an example workflow for the images retrieval:

docker run -d -p 5000:5000 --name registry registry
docker build -t localhost:5000/my-image my_project_dir
docker push localhost:5000/my-image

later you wil run : docker pull localhost:5000/my-image

You could also tag an existing image: docker tag my-image localhost:5000/my-image

5. Evaluation Criteria

  1. Functionality:

    • All routes must work as intended.
    • Application must persist data written via /write/{data} after container restarts.

  2. Accessibility:

    • Application must be accessible at http://localhost:8777 in both Docker Compose and Kubernetes.

  3. Scaling:

    • The application should scale between 1 and 5 replicas based on traffic, as observed during load testing.

  4. Correct Use of Tools:

    • Docker Compose must be used for local development.
    • Kubernetes must be used for production-like orchestration.
