TP : Entrypoint et Healthcheck avec Docker Compose

Objectif

Cet exercice adapte le TP précédent ("Conteneur Couteau Suisse") pour utiliser Docker Compose.

L'objectif est de :

  1. Comprendre comment exécuter des tâches ponctuelles (comme un outil en ligne de commande) définies comme des services Compose, en utilisant docker-compose run.
  2. Apprendre à définir ou surcharger un HEALTHCHECK directement dans le fichier docker-compose.yml.
  3. Observer l'état de santé d'un service géré par Compose.

Prérequis

  • Docker et Docker Compose installés.
  • Les Dockerfile.cowsay et Dockerfile.webhealth fonctionnels de l'exercice précédent (ou la capacité de les recréer).

À Faire

  1. Créer le fichier docker-compose.yml : Créez un nouveau fichier docker-compose.yml.

  2. Définir le service "Outil" (cowsay) :

    • Définissez un service (nommez-le cowsay_tool, par exemple) qui se base sur votre Dockerfile.cowsay. Utilisez la section build: appropriée (vous pouvez spécifier le context et le dockerfile).
    • Note : L'ENTRYPOINT est déjà défini dans votre Dockerfile.cowsay. Compose l'utilisera par défaut. Vous n'avez généralement pas besoin de le redéfinir ici sauf pour l'écraser.

  3. Exécuter le service comme une commande :

    • Pour lancer une tâche ponctuelle basée sur la définition d'un service, on utilise docker-compose run. Essayez : 
      # Lance cowsay avec un message
docker-compose run --rm cowsay_tool "Compose me fait parler !"

# Lance cowsay avec une option (-f)
docker-compose run --rm cowsay_tool -f tux "Linux et Compose !"
    • Observez le résultat. Le flag --rm est utile pour supprimer le conteneur juste après son exécution. Notez la différence avec docker-compose up.

  4. Définir le service Web avec Healthcheck :

    • Ajoutez un deuxième service à votre docker-compose.yml (nommez-le web, par exemple) qui se base sur votre Dockerfile.webhealth.
    • Vous pouvez laisser Compose utiliser le HEALTHCHECK défini dans le Dockerfile, OU (pour pratiquer la syntaxe Compose), vous pouvez le définir explicitement dans la section healthcheck: du service web dans votre docker-compose.yml. Essayez de reproduire les mêmes paramètres (interval, timeout, retries, start_period) que ceux définis dans le Dockerfile.webhealth.
    • (Indice : La syntaxe pour la commande de test dans healthcheck: est légèrement différente de celle du Dockerfile. Cherchez test dans la documentation healthcheck de Compose. Souvent, on utilise une liste comme test: ["CMD", "curl", "--fail", "http://localhost:8000/"] ou test: ["CMD-SHELL", "curl... || exit 1"]).
    • Assurez-vous de mapper le port 8000 si vous voulez vérifier avec votre navigateur (facultatif, docker ps suffit).

  5. Lancer et observer la santé :

    • Démarrez uniquement le service web (ou les deux si vous préférez) en arrière-plan : 
      docker-compose up -d web
# ou juste: docker-compose up -d
    • Utilisez docker-compose ps pour observer l'état du service web. Regardez la colonne State ou Status. Vous devriez voir son état de santé évoluer de starting à healthy (si tout va bien).
    • (Bonus) : Comme dans l'exercice précédent, essayez de simuler une panne (docker exec ... pkill python3) et observez le statut passer à unhealthy dans docker-compose ps après quelques tentatives.

  6. Arrêter et nettoyer : Utilisez docker-compose down à la fin.

À Méditer / Comparer

  • Quelle est la différence entre docker-compose up et docker-compose run ? Quand utiliseriez-vous l'un ou l'autre ?
  • Est-il préférable de définir ENTRYPOINT et HEALTHCHECK dans le Dockerfile ou dans docker-compose.yml ? Quels sont les avantages/inconvénients de chaque approche ? (Indice : Dockerfile définit l'image "par défaut", Compose permet de l'adapter/l'orchestrer).

À Vérifier (pour vous)

  • Avoir un docker-compose.yml définissant les deux services.
  • Pouvoir exécuter le service cowsay_tool avec des arguments via docker-compose run.
  • Pouvoir démarrer le service web avec docker-compose up et observer son état de santé devenir healthy via docker-compose ps.
