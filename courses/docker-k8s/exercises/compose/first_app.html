TP : Application Backend + DB avec Docker Compose

Objectif

Cet exercice consiste à reprendre l'application multi-conteneurs (Backend Python + Base de Données MariaDB/Mongo) réalisée précédemment avec des commandes docker run manuelles, et à la réimplémenter en utilisant Docker Compose.

L'objectif est de :

  1. Comprendre la structure d'un fichier docker-compose.yml.
  2. Définir et configurer plusieurs services (backend, base de données) de manière déclarative.
  3. Gérer le réseau, les volumes, les variables d'environnement et les dépendances entre services via Docker Compose.
  4. Apprécier la simplification apportée par Docker Compose par rapport aux commandes manuelles.

Prérequis

  • Docker et Docker Compose installés.
  • Le code source fonctionnel du backend Python (avec son Dockerfile) de l'exercice précédent.
  • Connaître les variables d'environnement nécessaires à la configuration de votre backend (pour la connexion à la base de données) et de votre base de données (utilisateur, mot de passe, nom de la base).

À Faire

  1. Créer le fichier docker-compose.yml : À la racine de votre projet backend, créez un fichier nommé docker-compose.yml.

  2. Définir la configuration : Remplissez ce fichier pour reproduire le déploiement de l'application (backend + base de données) tel qu'il fonctionnait avec les commandes docker run manuelles. Votre fichier docker-compose.yml doit respecter les points suivants :

    • Définir deux services : un pour la base de données (db) et un pour le backend (backend).
    • Le service db doit utiliser l'image officielle mariadb (ou mongo) appropriée.
    • Le service backend doit être construit à partir de votre Dockerfile existant.
    • Configurer le service db avec les variables d'environnement nécessaires (ex: MYSQL_ROOT_PASSWORD, MYSQL_DATABASE, etc. pour MariaDB). Consultez la documentation de l'image Docker Hub si besoin pour les noms exacts des variables.
    • Configurer le service backend avec les variables d'environnement requises pour se connecter à la base de données (ex: DB_HOST, DB_USER, DB_PASSWORD, DB_NAME). Utilisez le nom du service db comme hôte pour la connexion (DB_HOST=db).
    • Assurer la persistance des données de la base de données à l'aide d'un volume nommé Docker.
    • Définir et utiliser un réseau personnalisé pour que les deux services communiquent.
    • Mapper le port du service backend pour qu'il soit accessible depuis l'hôte sur le port 5000 (ou le port que votre backend utilise).
    • Utiliser depends_on dans le service backend pour indiquer sa dépendance vis-à-vis du service db.
    • (Bonne pratique) : Évitez de coder en dur les mots de passe dans le fichier docker-compose.yml. Recherchez comment Docker Compose peut utiliser un fichier .env pour gérer ce type d'informations sensibles.

  3. Lancer l'application : Depuis le répertoire contenant votre docker-compose.yml, utilisez la commande docker-compose up -d pour démarrer l'ensemble des services.

  4. Vérifier le fonctionnement :

    • Utilisez docker-compose ps pour voir l'état des conteneurs.
    • Consultez les logs avec docker-compose logs backend et docker-compose logs db.
    • Accédez à votre application via http://localhost:5000 (ou le port mappé) et testez les fonctionnalités (CRUD) pour confirmer que la communication avec la base de données est opérationnelle.

  5. Arrêter l'application : Utilisez la commande docker-compose down pour arrêter et supprimer les conteneurs et le réseau créés par Compose. Utilisez l'option -v si vous souhaitez également supprimer le volume de données.

À Méditer / Comparer

  • Analysez votre fichier docker-compose.yml. Quelle est sa complexité par rapport à l'ensemble des commandes docker run et docker network create que vous aviez utilisées précédemment ?
  • Quels sont les avantages que vous voyez à utiliser Docker Compose pour ce scénario ?

À Vérifier (pour vous)

  • Avoir un fichier docker-compose.yml valide qui décrit l'application.
  • Pouvoir démarrer l'ensemble avec docker-compose up -d.
  • L'application backend est accessible et fonctionnelle (communication avec la DB OK).
  • Pouvoir arrêter proprement avec docker-compose down.
