TP : Communication Client/Serveur avec Docker Compose

Objectif

Cet exercice vise à reprendre le scénario simple "Client/Serveur" (deux conteneurs communiquant sur un réseau personnalisé) et à l'implémenter avec Docker Compose.

L'objectif est de :

  1. Définir deux services simples (un serveur minimaliste, un client) dans un fichier docker-compose.yml.
  2. Comprendre comment Docker Compose gère la création du réseau et permet la communication entre services en utilisant leurs noms.
  3. Comparer la simplicité de l'approche Compose par rapport aux commandes docker network create et docker run manuelles de l'exercice précédent.

Prérequis

  • Docker et Docker Compose installés.
  • Le code source et les Dockerfile fonctionnels pour le serveur HTTP minimal et pour le client (celui qui utilise curl, wget ou un script pour contacter le serveur) de l'exercice "Deux conteneurs qui se parlent (sans docker-compose)".
  • Assurez-vous que le client est configuré pour contacter le serveur via un nom d'hôte (ex: http://server:5000/). Si ce nom était passé via une variable d'environnement (SERVER_URL), c'est encore mieux.

À Faire

  1. Créer le fichier docker-compose.yml : Dans un répertoire contenant les contextes de build (ou les Dockerfiles) pour votre serveur et votre client, créez un fichier docker-compose.yml.

  2. Définir la configuration : Remplissez ce fichier pour lancer les deux conteneurs (serveur et client) et leur permettre de communiquer. Votre fichier docker-compose.yml doit :

    • Définir deux services, par exemple nommés server et client.
    • Chaque service doit être construit à partir de son Dockerfile respectif.
    • Assurez-vous que le client contacte le serveur en utilisant le nom du service défini dans ce fichier (ex: l'URL http://server:5000/ si le service serveur est nommé server). Si votre client utilise une variable d'environnement pour l'URL du serveur, définissez cette variable dans la section environment du service client.
    • Utilisez depends_on dans le service client pour indiquer qu'il dépend du service server.
    • (Facultatif mais recommandé pour la clarté) : Définir explicitement un réseau de type bridge et connecter les deux services à ce réseau (même si Compose en créerait un par défaut).
    • (Info) : Vous n'avez normalement pas besoin de mapper de port (ports:) pour le service server si seul le conteneur client doit y accéder. Le mapping de port n'est utile que pour accéder au service depuis l'hôte.

  3. Lancer les services : Depuis le répertoire contenant votre docker-compose.yml, lancez les services avec :

    docker-compose up
    • Observez les logs affichés. Vous devriez voir le serveur démarrer, puis le client démarrer, effectuer sa requête et afficher la réponse du serveur ("Hello client" ou similaire).
    • Si votre client est un script qui s'exécute une fois puis se termine, le conteneur client s'arrêtera (vous verrez client_container_name exited with code 0), mais docker-compose up continuera à tourner à cause du serveur. Vous pouvez l'arrêter avec Ctrl+C.

  4. Vérifier (alternative) : Si vous lancez en mode détaché (docker-compose up -d), vous pouvez voir les logs après coup avec docker-compose logs client et docker-compose logs server.

  5. Arrêter et nettoyer : Utilisez docker-compose down pour arrêter et supprimer les conteneurs et le réseau.

À Méditer / Comparer

  • Comparez ce fichier docker-compose.yml aux commandes docker network create, docker run server..., docker run client... de l'exercice précédent.
  • Comment la création du réseau et la résolution du nom server par le client sont-elles gérées par Compose ? Qu'est-ce qui est implicite ?

À Vérifier (pour vous)

  • Avoir un fichier docker-compose.yml qui définit les services server et client.
  • En lançant docker-compose up, observer dans les logs que le client réussit à contacter le server par son nom et affiche la réponse attendue.
  • Pouvoir arrêter proprement avec docker-compose down. elui-là est plus simple que le précédent (Backend+DB), mais il illustre très bien la simplification du réseau et du lancement par Compose. Ça te semble correct comme niveau de guidage ?
