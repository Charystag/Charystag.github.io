TP : Configuration via Environnement avec Docker Compose (Mode Autonome)

Objectif

Cet exercice consiste à reprendre l'application configurée via variables d'environnement de l'exercice précédent et à la réimplémenter en utilisant Docker Compose. L'accent est mis sur votre capacité à trouver et appliquer les bonnes directives Compose pour gérer la configuration.

L'objectif est de :

  1. Structurer un fichier docker-compose.yml pour un service unique.
  2. Définir et surcharger des variables d'environnement en utilisant les mécanismes de Docker Compose (environment:, intégration de fichiers .env, variables du shell).
  3. Comparer l'approche Compose avec les options manuelles de docker run.

Prérequis

  • Docker et Docker Compose installés.
  • Le code source de votre application (qui lit APP_PORT, GREETING_MSG, API_KEY, etc. depuis l'environnement) et son Dockerfile (celui de l'exercice précédent, qui définit des ENV par défaut).
  • Le fichier .env créé lors de l'exercice précédent.

À Faire

  1. Créer le fichier docker-compose.yml : À la racine de votre projet, créez un fichier docker-compose.yml.

  2. Définir la configuration du service : Configurez un unique service (nommez-le app) dans ce fichier pour qu'il lance votre application. Ce service doit :

    • Être construit (build) à partir de votre Dockerfile existant.
    • Définir une variable d'environnement GREETING_MSG via la section environment: de Compose, avec une valeur différente de celle par défaut dans votre Dockerfile.
    • Mapper le port approprié (ports:) pour rendre l'application accessible depuis l'hôte.
    • (Indice : Cherchez la documentation pour services, build, ports, environment dans Docker Compose).

  3. Test 1 : Priorité environment: (Compose) sur ENV (Dockerfile)

    • Lancez le service : docker-compose up -d
    • Vérifiez : L'application utilise-t-elle le GREETING_MSG défini dans docker-compose.yml ? Quelles sont les valeurs pour APP_PORT et API_KEY (celles du Dockerfile ou autres) ? Utilisez http://localhost:[port_mappé] et docker-compose logs app.
    • Arrêtez : docker-compose down.

  4. Test 2 : Intégration du fichier .env

    • Placez votre fichier .env existant dans le même répertoire que docker-compose.yml.
    • Modifiez docker-compose.yml pour qu'il utilise les variables définies dans ce fichier .env. Vous devrez probablement utiliser la syntaxe de substitution de variables de Compose (ex: ${VARIABLE}). Assurez-vous que les valeurs de APP_PORT, GREETING_MSG et API_KEY proviennent maintenant du fichier .env lorsqu'elles y sont définies. Adaptez également le mapping de port si APP_PORT change via le .env.
    • (Indice : Cherchez comment Docker Compose charge automatiquement les fichiers .env et comment utiliser la substitution de variables ${...} dans votre docker-compose.yml).
    • Lancez le service : docker-compose up -d
    • Vérifiez : L'application utilise-t-elle bien les valeurs (APP_PORT, GREETING_MSG, API_KEY) définies dans votre fichier .env ? Vérifiez via le navigateur/curl et les logs.
    • Arrêtez : docker-compose down.

  5. Test 3 : Priorité Variable Shell sur .env / environment:

    • Dans votre terminal hôte, exportez une variable d'environnement (par exemple GREETING_MSG) avec une valeur encore différente.
    • Depuis ce même terminal, relancez Compose : docker-compose up -d
    • Vérifiez : Quelle valeur de GREETING_MSG l'application utilise-t-elle maintenant ? (Indice : Les variables définies dans le shell qui lance Compose ont généralement la priorité).
    • Nettoyez votre variable d'environnement hôte et arrêtez : docker-compose down.

À Méditer / Comparer

  • Comment la section environment: de Compose se compare-t-elle à l'instruction ENV du Dockerfile et à l'option -e de docker run ?
  • Comment l'utilisation automatique du fichier .env par Compose simplifie-t-elle la gestion des configurations par rapport à l'option --env-file de docker run ?
  • Quelle est la règle de priorité entre les différentes méthodes de définition des variables d'environnement (Shell hôte, .env, environment: dans Compose, ENV dans Dockerfile) ?

À Vérifier (pour vous)

  • Avoir un fichier docker-compose.yml valide qui répond aux exigences.
  • Avoir un fichier .env fonctionnel.
  • Pouvoir vérifier par les logs ou l'accès web que les variables sont correctement prises en compte et que les priorités sont respectées lors des différents tests.
