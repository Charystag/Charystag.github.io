TP : Config Ton Conteneur (Variables d'Environnement)

Objectif

Fini de coder en dur la config ! On va voir comment piloter un conteneur Docker de l'extérieur juste avec les variables d'environnement (ENV). Le but est de rendre tes images flexibles et réutilisables sans avoir à les reconstruire pour changer un port ou une clé API.

On va apprendre à :

  1. Faire lire à une appli ses paramètres depuis l'environnement.
  2. Définir des valeurs par défaut dans le Dockerfile via ENV.
  3. Écraser ces valeurs au lancement (docker run) avec l'option -e.
  4. Utiliser un fichier .env pour gérer plusieurs variables via --env-file.

Prérequis

  • Docker fonctionnel.
  • Une petite application web simple (peu importe le langage : Node.js, Python/Flask, Ruby/Sinatra, Go...) qui :
    • Sait démarrer un serveur HTTP.
    • Écoute sur un port réseau.
    • Répond à une requête GET sur la racine /.
  • Savoir comment lire des variables d'environnement dans le langage choisi (c'est ça l'étape clé du code de l'appli).

À Faire

  1. Prépare ton application :

    • Assure-toi que ton code lit ses paramètres de configuration depuis les variables d'environnement. Prévois des valeurs par défaut si une variable n'est pas définie.
    • Paramètres à lire (minimum) :
      • APP_PORT : Le port sur lequel ton serveur doit écouter.
      • GREETING_MSG : Un message qu'il affichera ou retournera.
      • API_KEY : Une clé API bidon pour l'exemple.
    • Exemples pour lire une variable VAR avec une valeur par défaut default :
      • Node.js : process.env.VAR || 'default'
      • Python : os.getenv('VAR', 'default') (nécessite import os)
      • (Adapte selon ton langage)

  2. Crée un Dockerfile simple :

    • Pars de l'image de base ubuntu:latest ou alpine:latest
    • Installe les outils pour developper dans ton langage
    • Mets en place la structure de base : WORKDIR, copie des fichiers nécessaires.
    • Copie le(s) fichier(s) de dépendances (ex: package.json, requirements.txt) et lance la commande pour installer les dépendances (npm install --production, pip install -r requirements.txt, bundle install, etc.).
    • Copie le reste du code source de ton appli.
    • Définis les ENV par défaut dans le Dockerfile avec ENV : 
      ENV APP_PORT=3000
ENV GREETING_MSG="Salut défini dans Dockerfile"
ENV API_KEY="default_key_from_dockerfile"
    • Expose le port en utilisant la variable : EXPOSE ${APP_PORT}
    • Définis le CMD pour lancer ton application (ex: CMD ["node", "server.js"], CMD ["python", "app.py"], CMD ["ruby", "app.rb"]).

  3. Build l'image : docker build -t mon-app-env .

  4. Test 1 : Valeurs par défaut

    • Lance le conteneur : docker run -d -p 8000:3000 --name test-default mon-app-env
    • Vérifie : Accède à http://localhost:8000 (ou via curl). Regarde aussi les logs avec docker logs test-default. Tu dois voir les valeurs par défaut définies dans le Dockerfile être utilisées.
    • Clean : docker rm -f test-default.

  5. Test 2 : Override avec -e

    • Lance en changeant le message et la clé API : docker run -d -p 8001:3000 --name test-e -e GREETING_MSG="Yo depuis le container" -e API_KEY="12345AZERTY" mon-app-env
    • Vérifie : Accède à http://localhost:8001 et check les logs. Le nouveau message et la clé API doivent apparaître.
    • Clean : docker rm -f test-e.

  6. Test 3 : Fichier .env

    • Crée un fichier texte nommé .env à côté de ton Dockerfile : 
      # .env
APP_PORT=9999
GREETING_MSG=Hello depuis .env !
API_KEY=cle_du_fichier_env
    • Lance avec ce fichier : docker run -d -p 9999:9999 --name test-envfile --env-file .env mon-app-env
    • Vérifie : Accède à http://localhost:9999 et check les logs. Les valeurs du fichier .env doivent avoir été prises en compte.
    • Clean : docker rm -f test-envfile.

  7. (Bonus) Priorité -e vs --env-file

    • Lance avec les deux, en mettant une valeur différente pour GREETING_MSG dans le -e par rapport au fichier .env : docker run -d -p 8888:9999 --name test-prio -e GREETING_MSG='Priorité à -e !!!' --env-file .env mon-app-env
    • Vérifie : Accède à http://localhost:8888. Quel message s'affiche ? (Normalement, -e gagne).
    • Clean : docker rm -f test-prio.

À avoir a la fin

  • Le code source de votre application (fichier principal, etc.).
  • Le fichier listant les dépendances (ex: package.json, requirements.txt).
  • Le Dockerfile.
  • Le fichier .env.
  • Les commandes docker build et docker run exactes que vous avez utilisées pour chaque test.
