Introduction au Langage UML

Diagramme 6 – Diagramme d’activité UML

Titre : Gestion d’un espace de coworking – côté administrateur

Consignes détaillées

1. Objectif du diagramme

Modéliser le processus métier de gestion des espaces (création, modification, désactivation) du point de vue d’un administrateur.
Le diagramme doit inclure les différents chemins possibles, les vérifications préalables, et les cas de conflit.

2. Éléments attendus dans le diagramme :

  • Nœud de départ clair : l’administrateur accède à l’interface de gestion des espaces
  • Bloc de décision initial : choix de l’action
    • [Créer un espace]
    • [Modifier un espace existant]
    • [Désactiver un espace]
Si création :
  • Remplissage du formulaire (nom, capacité, type, etc.)
  • Vérification de la complétude du formulaire
    • Si formulaire incomplet : message d’erreur, retour
    • Si formulaire valide : création de l’espace et confirmation
Si modification :
  • Sélection d’un espace existant
  • Modification des informations
  • Sauvegarde
  • Confirmation de modification
Si désactivation :
  • Vérification : l’espace a-t-il des réservations actives ?
    • Si oui : afficher un avertissement, bloquer ou proposer une replanification
    • Si non : désactivation effective et message de confirmation

3. Représentation obligatoire :

  • Nœud de fin clair
  • Blocs décisionnels :
    • Sélection de l’action
    • Vérifications de formulaire
    • Gestion de conflits liés aux réservations actives
  • Flux alternatifs :
    • Création échouée
    • Désactivation refusée
  • Actions nommées clairement : “Remplir le formulaire”, “Vérifier réservations actives”, “Afficher avertissement”, etc.
  • Possibilité d’interrompre le processus à tout moment (zone d’interruption optionnelle si bien maîtrisée)

4. Contraintes techniques :

  • Pas besoin de modéliser les échanges avec la base de données
  • L’accent est mis sur la logique métier de la gestion d’un espace
  • Vous ne devez modéliser que le point de vue de l’admin, pas les impacts côté client

Critères de notation :

  • Structure logique et complète du processus
  • Utilisation pertinente des blocs de décision
  • Modélisation claire des conflits et erreurs
  • Lisibilité et pertinence des actions
