Les différents types de redondance de stockage dans Azure

Azure offre plusieurs niveaux de redondance pour garantir la disponibilité, la durabilité et la récupération des données en cas de panne ou de sinistre. Ces niveaux de redondance permettent de choisir la protection adéquate en fonction des besoins de votre application et de votre budget.

1. Local Redundant Storage (LRS)

Description :

Le LRS (Local Redundant Storage) conserve trois copies de vos données dans un seul datacenter. Ces copies sont réparties sur des serveurs distincts dans un même centre de données pour protéger contre les défaillances matérielles locales (comme un disque défectueux).

Caractéristiques :

  • Données répliquées uniquement dans un datacenter unique.
  • Faible coût.
  • Pas de protection contre les sinistres impactant le datacenter entier (pannes électriques, inondations, incendies).

Cas d'usage :

  • Données qui peuvent être facilement recréées ou restaurées.
  • Applications locales ou de test.

👉 Documentation LRS

2. Zone Redundant Storage (ZRS)

Description :

Le ZRS (Zone Redundant Storage) conserve trois copies de vos données dans différentes zones de disponibilité au sein de la même région Azure. Une zone de disponibilité est un regroupement de datacenters physiquement séparés et indépendants.

Caractéristiques :

  • Haute disponibilité avec répartition des copies dans des zones distinctes.
  • Protection contre les pannes d’un datacenter ou d’une zone entière.
  • Latence légèrement accrue par rapport au LRS (en raison de la réplication entre zones).

Cas d'usage :

  • Données critiques nécessitant une disponibilité élevée.
  • Applications qui doivent rester en ligne même en cas de panne d'une zone Azure.

👉 Documentation ZRS

3. Geo-Redundant Storage (GRS)

Description :

Le GRS (Geo-Redundant Storage) conserve six copies de vos données : trois dans un datacenter principal (LRS) et trois dans un datacenter secondaire situé dans une autre région Azure distante (réplication géographique).

En cas de sinistre majeur dans la région principale, les données peuvent être lues depuis la région secondaire.

Caractéristiques :

  • Protection contre des sinistres régionaux (comme une panne ou une catastrophe naturelle).
  • La région secondaire est en lecture seule jusqu’à une bascule manuelle.
  • Latence accrue pour la réplication inter-régions.

Cas d'usage :

  • Données critiques nécessitant une haute durabilité.
  • Applications avec tolérance aux sinistres régionaux.

👉 Documentation GRS

4. Geo-Zone Redundant Storage (GZRS)

Description :

Le GZRS (Geo-Zone Redundant Storage) combine les avantages du ZRS et du GRS. Les données sont répliquées dans des zones de disponibilité au sein de la région principale, et également répliquées dans une région secondaire distante.

Caractéristiques :

  • Haute disponibilité au sein de la région principale grâce au ZRS.
  • Protection contre des sinistres régionaux grâce à la réplication géographique.
  • Résilience maximale.

Cas d'usage :

  • Données critiques et stratégiques nécessitant une protection maximale.
  • Applications hautement disponibles, même en cas de sinistre régional.

👉 Documentation GZRS

5. Read-Access Geo-Redundant Storage (RA-GRS)

Description :

Le RA-GRS (Read-Access Geo-Redundant Storage) est une variante du GRS qui permet un accès en lecture seule aux données répliquées dans la région secondaire.

Caractéristiques :

  • Ajoute la capacité d'accès en lecture à partir de la région secondaire sans bascule.
  • Utilisé pour augmenter la disponibilité globale et réduire la latence pour des utilisateurs situés près de la région secondaire.

Cas d'usage :

  • Données consultées fréquemment dans plusieurs régions.
  • Réduction des temps d'arrêt pour les applications basées sur les données.

👉 Documentation RA-GRS

6. Read-Access Geo-Zone Redundant Storage (RA-GZRS)

Description :

Le RA-GZRS est une variante du GZRS qui ajoute un accès en lecture seule à la région secondaire, comme le RA-GRS.

Caractéristiques :

  • Réplication multi-zones dans la région principale (ZRS).
  • Réplication géographique dans une région secondaire avec accès en lecture.
  • Résilience et disponibilité maximales.

Cas d'usage :

  • Applications critiques nécessitant à la fois une haute disponibilité locale et une récupération géographique rapide.
  • Réduction des temps d'accès en lecture pour des utilisateurs globaux.

👉 Documentation RA-GZRS

Bonnes pratiques pour choisir une option de redondance

  1. Besoins de disponibilité :

    • Si vous n’avez pas besoin de haute disponibilité, LRS peut suffire.
    • Pour des applications critiques, préférez ZRS ou GZRS.

  2. Besoins de récupération après sinistre :

    • Pour garantir une récupération en cas de panne régionale, choisissez GRS, GZRS, ou leurs variantes RA.

  3. Budget :

    • Les solutions géographiquement redondantes (GRS, GZRS) sont plus coûteuses, mais elles offrent une meilleure résilience.

  4. Accès global :

    • Si vous avez des utilisateurs répartis sur plusieurs régions, optez pour RA-GRS ou RA-GZRS pour une meilleure expérience utilisateur.
