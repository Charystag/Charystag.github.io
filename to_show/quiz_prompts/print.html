Question Rules

  1. Adhere to Parameters: The question must align with every parameter specified, including audience, background, topic, difficulty, type, programming language and purpose. All these parameters will be provided in the rules object.

  • If programming_language is specified (e.g., "Python", "Java", "AWS"), the question must use concepts, syntax, or terminology relevant to that language or platform.

  • If programming_language is null, the question must remain language-agnostic. Do not rely on any language-specific features or syntax.

  • If topic is specified, the question must be clearly relevant to that topic.

    • Relevance: The question should directly involve concepts, features, operations, or scenarios commonly associated with the specified topic. For example, if the topic is "Loops" and the language is Python, the question should involve iteration structures recognizable in Python, not unrelated tasks. If the topic is "S3" and the language is AWS, the question should involve scenarios related to S3 buckets or objects, not something unrelated like EC2 instances.

  • If topic is null and programming_language is not null, select a concept or scenario inherently related to that language or platform. For instance, if programming_language = "Python" and topic = null, you might choose a fundamental Python concept that matches the given audience, difficulty, and purpose, such as basic data types or simple loops for beginners, or function usage for intermediate.

  • If both topic and programming_language are null, then any general programming question aligned with the other parameters is acceptable (though this scenario indicates a potential input error). In this case, choose a question that suits the specified audience, difficulty, and purpose in a general, language-neutral context.

  1. Uniqueness: The question must not duplicate or closely resemble any of the previously listed previous_questions.
    • Closely Resembling: A question is considered “closely resembling” if it:
      • Uses the exact same code snippet or only trivial modifications (e.g., changing variable names without altering the concept tested).
      • Tests the exact same concept in the same manner (e.g., previously asked the output of print(2+2); now asking print(1+3) would be too similar as it tests the same trivial arithmetic output without introducing a new concept).
    • The new question should introduce at least one distinct element, concept, or scenario not previously tested.
  2. Focus on Novelty: The question should present at least one new aspect or variation. For example, if previous questions tested basic arithmetic, a novel question might incorporate a loop or a function call to encourage the learner to apply their knowledge differently.
  3. Encourage Learning: The question should help expand the learner’s understanding. It should not merely re-test an identical concept at the same level. Instead, it might add a slightly more complex scenario or build upon previously tested knowledge.
  4. No Answers/Explanations: The output should only contain the generated question and its type, no solutions or hints.

Instructions:

  • Answer Options:

    • For multiple_choice: Provide exactly 4 answer options by default, unless the difficulty or concept strongly suggests adding a 5th option. Exactly one answer must be marked as correct.
    • For true_false: Provide two options, "True" and "False," with exactly one marked as correct.

  • Explanation:
    Provide a concise explanation that justifies why the correct answer is correct. Tailor the depth and complexity of the explanation to the specified audience, background, and difficulty. For conceptual questions, clarify the theory or syntax. For problem-solving questions, walk through the reasoning steps. For implementation questions, highlight key practical details or code elements.

  • Documentation Links:
    If directly relevant and high-quality official documentation exists for the tested concept, include up to 3 links. If no straightforward, suitable official documentation is available, omit links.

Parameter Definitions

  • Audience: Describes the skill level and profile of the target learner.

    • Beginner: Simple, direct, no assumptions. Introduce the most basic concepts with clarity.
    • Intermediate: Build on fundamentals. The learner can interpret and apply basic concepts, handle simple problem-solving, and understand common terminology.
    • Advanced: Complex, multi-concept, potentially tricky scenarios requiring deep understanding and analysis. The learner can handle subtlety, debugging, and less common features.

  • Background: Provides context about the learner’s environment or prior knowledge.

    • Example: "No prior programming experience" means you must avoid assuming they know any programming terms. "Familiar with Python basics" allows using Python syntax and basic constructs.

  • Programming Language: Specifies the main language or technology context for the question.

    • Example: "Python", "Java", "AWS", "Azure", or null if the question should be language-agnostic.
    • If a programming language or platform is specified, the question should use relevant syntax, terminology, or services. If null, the question remains conceptually focused without language-specific syntax.

  • Topic: The subject matter or concept the question focuses on. This can be a language feature (e.g., Loops, Data Structures), a platform-related subject (e.g., S3, EC2, Virtual Networks), or more general programming concepts.

  • If a specific topic is provided, the question should directly relate to it.

  • If null, then the question should remain broadly relevant to the chosen programming language or context without focusing on a particular subtopic.

  • Difficulty: Indicates the complexity level of the question. The question must match the specified difficulty.

    • Easy: Simple recall or recognition tasks suitable for complete beginners. Typically involves direct, obvious outcomes (e.g., output of print(2+3)).
    • Medium: Requires applying known concepts in a straightforward scenario.
      • Straightforward means no hidden tricks, obscure language features, or unexpected behavior.
      • Example: If the learner knows loops and arithmetic, a medium question might ask: "What does the following loop print?", where the loop and arithmetic are directly understandable without guessing unusual behavior.
      • Non-straightforward (which should NOT be in medium) might involve subtle language quirks, odd data structures rarely encountered, or performance optimizations not expected at this level.
    • Hard: Demands advanced problem-solving. This might involve multiple concepts together, require understanding edge cases, debugging code, or interpreting less common language features. For example, analyzing nested loops with conditional logic that produce a non-obvious result, or identifying why a code snippet raises a TypeError due to a subtle misuse of types.

  • Type: Specifies the format of the question.

    • Multiple Choice: The final output is a question expecting multiple-choice answers. Do not provide the answers, only the question prompt.
    • True/False: The final output is a statement where the student decides if it’s true or false. Only output the statement.

  • Purpose: Defines the intent of the question.

    • Conceptual: Tests definitions, theory, or syntax without requiring execution or problem-solving. (e.g., "Which of these keywords defines a function in Python?")
    • Problem Solving: Requires applying concepts to derive an answer. (e.g., "What will be printed by this loop?")
    • Implementation: Focuses on real-world coding tasks, debugging, or understanding practical use cases. (e.g., "Which code snippet correctly initializes a Flask route?")

Examples

Example 1: Beginner, Conceptual, Python Data Types

Input:

{
  "rules": {
    "audience": "beginner",
    "background": "No prior programming experience.",
    "programming_language": "Python",
    "topic": "Data Types",
    "difficulty": "easy",
    "type": "multiple_choice",
    "purpose": "conceptual"
  },
  "previous_questions": [
    "What is the output of print(2 + 2)?",
    "Which of these is a valid Python variable name?"
  ]
}

Rationale:

  • Audience & Background: Beginner with no experience → stick to very basic concepts.
  • Programming Language: Python → use Pythonic terminology.
  • Topic: Data Types → introduce Python data types, a fundamental concept.
  • Difficulty: Easy → simple recall or recognition, no tricky aspects.
  • Purpose: Conceptual → testing understanding of what data types exist in Python.
  • Novelty: Previously tested arithmetic and variable naming; now test recognition of a specific data type form.

Output:

{
  "question": "Which of the following represents a string data type in Python?",
  "type": "multiple_choice"
}

This question shifts focus from arithmetic and variable names to identifying a data type (a string), encouraging learning and variety.
